# 學生作業 AI 評分與即時進度燈號顯示系統規格書

- 版本：V1.0
- 日期：2026-04-19
- 適用範圍：第一版

## 1. 專案目標

本系統用於課堂中的即時練習情境，讓學生提交 Arduino 積木程式 XML 作業後，由系統自動完成結構解析、規則判斷、AI 回饋生成，並以燈號與等第呈現學生學習狀態。老師可即時掌握全班各作業進度，並於課後查看統計結果與人工覆核評分。

## 2. 專案範圍

- 第一版僅支援單一班級、單一老師。
- 使用情境以實體課堂中的即時練習為主。
- 老師與學生皆需帳號密碼登入。
- 老師可預先建立多堂課堂活動。
- 每堂課堂活動可指定多份作業。
- 學生登入後先查看作業列表，再選擇作業提交。

## 3. 使用者角色

| 角色 | 說明 | 主要權限 |
|---|---|---|
| 老師 | 單一班級授課教師 | 建立課堂活動、指定作業、查看班級總覽、查看歷史提交、查看統計報表、人工改判 |
| 學生 | 班級學生 | 查看作業列表、提交 XML、查看自己的結果、查看全班燈號進度 |

## 4. 使用情境

### 4.1 老師端

- 上課前建立或選擇一堂課堂活動。
- 指定本次活動包含的多份作業。
- 上課中即時查看每位學生對每份作業的提交狀態與燈號。
- 點擊燈號查看學生該次作業的 AI 回饋內容。
- 課後查看每份作業的燈號分布、等第分布與歷史提交。
- 必要時人工覆核並修改最終燈號與等第。

### 4.2 學生端

- 登入後查看作業列表。
- 選擇某份作業，貼上 Arduino 積木 XML。
- 按下提交後等待系統非同步評分。
- 查看最新一次的等第、燈號與 AI 回饋。
- 查看全班燈號進度，但只能查看自己的回饋內容。

## 5. 功能需求

### 5.1 登入與權限

- FR-01：老師需以帳號密碼登入。
- FR-02：學生需以帳號密碼登入。
- FR-03：學生僅能查看自己的 AI 回饋內容。
- FR-04：學生可查看全班燈號進度，但不可查看其他學生的回饋內容。

### 5.2 課堂活動與作業管理

- FR-05：老師可預先建立多堂課堂活動。
- FR-06：每堂課堂活動可指定多份作業。
- FR-07：老師可選擇開啟其中一堂課堂活動。
- FR-08：學生登入後須先看到作業列表，再依序選擇作業。

### 5.3 作業提交

- FR-09：學生每次提交僅對應一份作業與一段 XML。
- FR-10：學生以貼上 Arduino 積木 XML 的方式提交作業。
- FR-11：系統需先驗證 XML 格式，格式錯誤不得進入評分流程。
- FR-12：學生可重複提交同一份作業。
- FR-13：系統需保留每次提交紀錄與提交時間。
- FR-14：歷史提交紀錄僅供老師查看。
- FR-15：每筆提交需自動綁定學生身份、當前課堂活動與所選作業。

### 5.4 評分與回饋

- FR-16：系統採非同步評分。
- FR-17：系統須先解析 XML 為積木結構。
- FR-18：系統須先進行規則式初判，再交由 AI 產生回饋。
- FR-19：規則式初判需檢查基本結構、關鍵積木、流程順序。
- FR-20：缺少必要積木或流程明顯錯誤時，直接判定為紅燈 / 待加強。
- FR-21：規則式初判與 AI 判斷衝突時，以規則式初判為準。
- FR-22：AI 回饋格式固定為做對的地方、問題摘要、修改建議。
- FR-23：學生端與老師端看到的回饋內容相同。

### 5.5 即時顯示

- FR-24：學生提交後，學生端先顯示處理中。
- FR-25：評分完成後，學生端自動更新結果。
- FR-26：老師端班級總覽須在每位學生每份作業評分完成時立即更新。
- FR-27：評分中狀態須顯示「評分中」。
- FR-28：評分失敗時，老師端顯示「評分失敗」，學生端顯示失敗訊息並允許重送。

### 5.6 課後分析與人工改判

- FR-29：老師課後可查看每份作業的燈號分布。
- FR-30：老師課後可查看每份作業的等第分布。
- FR-31：老師可人工修改最終燈號與等第。
- FR-32：系統須同時保留 AI 原始結果與老師最終結果。
- FR-33：班級總覽須標示該筆結果已由老師修正。
- FR-34：學生端只顯示最終結果，不顯示老師已修正標記。

## 6. 輸入規格

| 項目 | 規格 |
|---|---|
| 輸入來源 | 學生手動貼上 XML |
| 資料格式 | Arduino 積木程式 XML |
| 編輯器來源 | 固定同一套積木編輯器 |
| 作業選擇 | 學生先選作業，再貼 XML |
| 單次提交內容 | 一份作業對應一段 XML |
| 驗證方式 | 先做 XML 格式驗證 |

## 7. 輸出規格

### 7.1 等第與燈號

| 等第 | 燈號 |
|---|---|
| 優 | 綠 |
| 良 | 藍 |
| 可 | 黃 |
| 待加強 | 紅 |

### 7.2 學生端輸出

- 最新一次提交的最終結果
- 等第
- 燈號
- AI 回饋文字
- 失敗時的錯誤訊息

### 7.3 AI 回饋格式

- 做對的地方
- 問題摘要
- 修改建議

### 7.4 老師端輸出

- 班級總覽
- 每位學生每份作業的提交狀態
- 每位學生每份作業的燈號
- 個別學生回饋內容
- 歷史提交紀錄
- 每份作業的燈號分布
- 每份作業的等第分布

## 8. 評分規則

### 8.1 評分依據

- 評分依據為老師預先設定的作業文字說明。
- 第一版不使用標準答案。
- 第一版不要求老師設定額外評分面向。

### 8.2 評分流程

1. 驗證 XML 格式。
2. 解析 XML 為積木結構。
3. 規則引擎執行初步判斷。
4. AI 根據作業說明與解析結果生成回饋與中間等級判斷。
5. 系統依規則優先原則產出 AI 原始結果。
6. 如老師人工改判，則產生老師最終結果。

### 8.3 規則式初判

規則式初判至少檢查以下項目：

- 基本結構是否合理
- 是否包含關鍵積木
- 積木流程順序是否合理

### 8.4 等第判定原則

- 綠燈 / 優：結構合理、關鍵積木齊全、流程正確、完整達成作業目標。
- 藍燈 / 良：已大致完成作業目標，仍有可改善之處。
- 黃燈 / 可：結構正確但完成度不足，由 AI 進一步判定。
- 紅燈 / 待加強：缺少必要積木或流程明顯錯誤。

### 8.5 衝突處理

- 當規則式初判與 AI 判斷不一致時，以規則式初判為準。

## 9. 系統流程

1. 老師登入。
2. 老師建立或選擇課堂活動。
3. 老師開啟本次課堂活動並指定多份作業。
4. 學生登入。
5. 學生查看作業列表並選擇作業。
6. 學生貼上 XML 並提交。
7. 系統驗證 XML 格式。
8. 若格式錯誤，回傳錯誤訊息，學生修正後重送。
9. 若格式正確，建立提交紀錄並顯示評分中。
10. 系統解析 XML 結構。
11. 規則引擎進行初判。
12. AI 根據作業說明與解析結果生成回饋與等第建議。
13. 系統產出 AI 原始結果。
14. 學生端自動更新最新結果。
15. 老師端班級總覽即時更新對應格子。
16. 若處理失敗，學生端顯示失敗訊息，老師端顯示評分失敗。
17. 課後老師查看統計結果與歷史提交。
18. 必要時老師人工修正最終燈號與等第。
19. 系統保留 AI 原始結果與老師最終結果。
20. 班級總覽標示該筆結果已由老師修正。

## 11. 非功能需求

- NFR-01：老師端班級總覽需支援即時更新。
- NFR-02：學生端提交後不得要求手動刷新才能看到結果。
- NFR-03：系統需保留提交歷史與改判紀錄以供追蹤。
- NFR-04：系統須區分老師與學生權限，避免學生讀取他人回饋。
- NFR-05：系統需清楚區分未提交、評分中、評分失敗、已完成、已修正等狀態。

## 12. 技術決策

- 前端部署平台使用 Vercel Hobby。
- 前端框架使用 React + TypeScript。
- 後端框架使用 FastAPI。
- 後端部署平台使用 Render Free Web Service。
- 正式資料庫使用 Supabase Free 提供的 Postgres。
- 驗證與登入能力優先整合 Supabase Free 的 Auth。
- 即時更新機制優先使用 Supabase Free 的 Realtime，必要時可搭配 WebSocket。
- LLM 服務使用 Gemini Developer API。
- 第一版評分模型使用 gemini-2.5-flash-lite。
- 第一版非同步任務先採用 FastAPI 內建 background task 或應用內 job queue。
- 第一版不先導入 Celery / Redis。

## 13. 第一版限制

- 僅支援單一班級、單一老師。
- 僅支援固定來源的 Arduino 積木 XML。
- 不支援圖片上傳。
- 不支援多種程式語言。
- 不支援學生查看歷史版本差異。
- 不支援學生查看其他同學的 AI 回饋內容。
